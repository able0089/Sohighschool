## Problem
The Railpack build fails during `pnpm install --frozen-lockfile` with error `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH`: "Cannot proceed with the frozen installation. The current 'overrides' configuration doesn't match the value found in the lockfile." The SNAPSHOT_CODE stage completed successfully, but BUILD_IMAGE fails immediately when pnpm attempts to install dependencies. The `overrides` field in `pnpm-workspace.yaml` contains extensive platform-specific package exclusions and an esbuild version pin that are not reflected in the current `pnpm-lock.yaml`. The `--frozen-lockfile` flag used by Railpack prevents pnpm from updating the lockfile during CI builds, so the mismatch causes an immediate failure.

## Diagnosis
The `pnpm-workspace.yaml` has an `overrides` section that excludes non-linux-x64 platform binaries for esbuild, lightningcss, @tailwindcss/oxide, rollup, and @expo/ngrok-bin, plus pins esbuild to version 0.27.3 and replaces @esbuild-kit/esm-loader with tsx. These overrides were added or modified without regenerating the lockfile afterward. When Railpack runs `pnpm install --frozen-lockfile`, pnpm detects that the overrides in the workspace config don't match the overrides hash stored in the lockfile and refuses to proceed. The triggering commit only changed `artifacts/api-server/src/bot/pokemon-bot.ts` â€” no dependency or workspace config files were touched â€” confirming the mismatch predates this commit and has caused at least four consecutive deployment failures.

## Fix
This fix requires running a command locally and cannot be implemented via file edits alone:

1. Clone the repository and checkout the `main` branch.
2. Run `pnpm install` at the repo root (no flags) â€” this will regenerate `pnpm-lock.yaml` to match the current `overrides` in `pnpm-workspace.yaml`.
3. Commit the updated `pnpm-lock.yaml` and push to `main`.
4. Railway will automatically trigger a new deployment because `/pnpm-lock.yaml` is now included in the service's watch patterns.
